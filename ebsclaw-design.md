# High-level data flow

```
Messaging apps (WhatsApp/Telegram/Discord/iMessage/...)
        │
        ▼
Gateway (single always-on process)
  - channel ingress/egress
  - input sanitization (strip metadata, control chars)
  - routing + policy checks
  - sessions (disk)
  - agent turns → block reply pipeline
  - tool calls
        │
        ▼
AI provider(s) (Anthropic/OpenAI/etc.) OR local model endpoint
        │
        ▼
Back to the channel
```

# Gateway 关键设计

- **plugin system** — 通过插件可以进行系统能力的扩展，例如支持 WhatsApp/Telegram/Discord 各类 IM 的功能就是通过插件的方式提供
- **channel** — 支持各类 IM、以及 Git Webhook、监控系统
  - Git Webhook 用于 PR/Commit 触发构建测试
  - 监控系统用于 Prometheus/Grafana 告警接入
- **terminal UI**
  - Gateway 模式：TUI 是 gateway 的下游客户端，通过 WS RPC 通信
  - Embedded 模式（默认）：TUI 是 gateway 的子集——复用了 gateway 的逻辑，但不启动 gateway 的网络服务层
- **cron 调度器** — 用确定性抖动防雷群、最小间隔防热循环、指数退避防重试风暴、三层超时防卡死、双文件分离保持久化性能

# Agent Runtime 关键设计

信任模型的推理能力，保持应用层框架尽可能简单

## System Prompt 构建

工作空间配置文件：

- `AGENTS.md` — 核心指令，定义 Agent 能做什么、不能做什么（基线规则）
- `MEMORY.md` — 文件作为记忆的索引

动态上下文（每轮对话实时组装）：

- 会话历史：最近的对话记录
- 技能文件（Skills）：特定任务的操作指南
- 记忆搜索结果：语义相关的历史对话，使用 LLM 的能力来进行语义搜索
- 工具输出结果

## 上下文压缩机制

六级线性排列、逐级升级，不是互斥选择。每级只处理自己负责的范围，解决不了才升级到下一级。

渐进损失原则：从"删工具结果"到"删消息段"到"全文摘要"，损失递增，不到万不得已不做高损失操作。

```
   请求到达

       ▼
```

## Skill 系统

Skill 系统是一个文档驱动的指令注入框架。Skill 不是可执行代码，而是由 SKILL.md 承载的结构化指令集，以 XML 目录形式注入 LLM system prompt，使用渐进式披露的策略，LLM 按需通过 read 工具加载完整指令后执行

SKILL.md 规范文件结构：

```
skill-name/
├── SKILL.md          (必须 — YAML frontmatter + Markdown 指令)
├── scripts/          (可选 — 可执行辅助脚本)
├── references/       (可选 — 按需加载的文档)
├── templates/        (可选 — 输出模板)
└── assets/           (可选 — 图片等资源)
```

## 记忆系统

### 自动记忆系统

- 作为 memory-core 插件子功能在 gateway 启动时注册到 gateway 中
- 设置 cron 调度器每隔 6 小时执行
- 触发线 = 上下文窗口 - 13000 Token 缓冲
- 独立记忆提取 agent，后台静默提取以下四类记忆：
  - **user 记忆** — 你是谁，你的身份、偏好、风格
  - **feedback 记忆** — 你纠正过它什么，错误、建议、成功经验
    - 不光要纠正，还要认可，如果只纠正，就会远离那些验证过的好方法
  - **project 记忆** — 项目的上下文、背景、代码结构
    - 凡是有时间，必须用绝对时间
  - **reference 记忆** — 外部资源链接、文档、历史记录
    - Agent 不需要知道外部系统的具体内容，只需要知道去哪里找
- 不记录可以从当前代码推导出来的信息，例如 Git 历史和最近的改动，代码结构
- 不记录当前会话结束就失效的信息

### AutoDream 系统

- 作为 memory-core 插件子功能在 gateway 启动时注册到 gateway 中
- 设置 cron 调度器每天凌晨 2 点执行
- 维护记忆的准确性，记忆要不断地更新，去掉过时的，保留重要的
- 记忆整理策略：
  - 定向：搞清楚自己现在知道什么
  - 收集：扫描今日日志，找值得留下的新消息
  - 整合：把新信息写进记忆文件
    - 相对时间改为绝对时间
    - 新发现推翻旧记忆直接修正，不留两条矛盾的记忆
    - 某条记忆当时写的是对的，但是现在情况变了，这条记忆不准确了，找出来并且修正
  - 剪枝：忘掉一些东西，省记忆空间
    - 更新索引
    - 删掉过时的
    - 压缩冗长的
    - 保持在 200 行以内

### 索引 + 独立文件存储记忆

每条记忆存为一个独立的 `.md` 文件，文件开头有一段 YAML 格式的元信息，包含三个字段：

- `name` — 人类可读的标识
- `description` — 一句话摘要，专门用于检索时的相关性匹配
- `type` — 标记四类型之一

示例：

```yaml
---
name: no-mock-database
description: 集成测试必须使用真实数据库，不能用 mock
type: feedback
---

集成测试必须使用真实数据库，不能用 mock。

**Why:** 上季度 mock 测试全部通过但生产环境迁移失败了。
**How to apply:** 在这个模块写测试时，始终连接真实数据库。
```

`MEMORY.md` 文件作为索引，它是一个不超过 200 行、每个文件不超过 25KB 的轻量目录，例如：

```markdown
- [No Mock Database](feedback_no_mock_db.md) — tests must use real DB
- [User Preferences](user_preferences.md) — prefers terse responses
- [Auth Rewrite](project_auth_rewrite.md) — driven by compliance, not tech debt
```

检查行数和字节数两个维度，超过就截断并警告

## 记忆召回

使用 LLM 进行语义检索，提取文件开头那段元信息里的 `name`、`description`、`type`，但不读取记忆的完整内容
